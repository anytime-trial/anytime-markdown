## 手順0: 前日比の重複除外（継続性）

調査の前に、`<docsRoot>/report/daily-research/` 配下の**直近3本**（今日より前の日付）の daily-research レポートを Read で読み、「既に報告済みのトピック・記事・新語彙」を把握する（各レポート末尾の「今日登場した新概念・新語彙」表と見出しをとくに確認）。調査では既報項目を再掲せず、**新たな更新のみ**を拾う（既報の進展・続報は「続報」と明記して可）。

ローカルにレポートが無い場合（fresh checkout・docs リポジトリ未取得・cloud/headless 実行等）は、S3 から本文を取得できる get_report を使う（fileName 指定。両経路ともデプロイ済み・稼働確認済み）:

- `mcp__mcp-cms__get_report` — ローカル mcp-cms 経由
- `mcp__claude_ai_mcp-cms-remote__get_report` — リモート mcp-cms-remote 経由（cloud/headless で有効）

## 調査方針

**調査対象期間: 調査実行日から過去2日間。** それ以前は原則対象外だが、未報かつ高影響度の項目は日付を明記の上で例外掲載可。

**一次情報を優先する。** 以下の優先順位で情報源を選定すること。

1. **公式ソース**: リリースノート、Changelog、GitHub Releases/PR/Issues、公式ブログ・ドキュメント
2. **当事者の発信**: 開発者本人のブログ・X（Twitter）・Hacker News・Bluesky での投稿、カンファレンス登壇資料
3. **実践者の一次体験**: 自身のプロダクトで実際に使った経験に基づくテックブログ・個人ブログ
4. **二次情報（参考程度）**: まとめ記事、ニュースサイトの要約、他者の体験の引用のみの記事

二次情報のみで構成された記事は、一次情報が見つからない場合の補足としてのみ採用する。

### 必読ソース（毎回必ず確認）

取りこぼしを防ぐため、毎回必ず以下をチェックする:

- **Claude Code 公式 changelog**（https://code.claude.com/docs/en/changelog）— 新バージョン・新プリミティブ。新バージョンがあればトピック2に必ず含める
- **Anthropic 公式ブログ**（https://www.anthropic.com/news）— モデル・料金・API 変更
- Hacker News の Claude Code / AI コーディング関連スレッド、Reddit r/ClaudeAI

### 影響度優先ルール

当プロジェクト（Tiptap ベース markdown エディタ）への影響度が「高」の項目——特に **トークンコスト・課金・Claude Code 本体の仕様変更・サブエージェント・Tiptap/エディタ関連**——は、件数枠に関わらず**最優先で必ず拾う**。趣味深いが影響度低の話題より、地味だが高影響の変更（課金体系・クレジット枠・changelog の破壊的変更等）を優先する。

## 調査トピック

### 1. Claude Code 活用の海外 SNS 投稿・知見（英語メイン） — 5件以上掲載
- CLAUDE.md 設計、ワークフロー設計、コンテキストマネジメント、スペック駆動開発、スキル設計、サブエージェント運用等に関する海外 SNS 上の投稿・議論
- **調査対象 SNS（英語圏を優先）:**
  - X（Twitter）: 「Claude Code」「CLAUDE.md」「claude code skills」「claude code subagent」等で英語投稿を検索
  - Bluesky: 同上のキーワードで英語投稿を検索
  - Hacker News: Claude Code 関連スレッド・コメント
  - Reddit: r/ClaudeAI, r/programming, r/LocalLLaMA 等の関連投稿
  - Discord: Claude / Anthropic 公式・非公式サーバー、AI 開発者コミュニティ（Cursor, AI Engineers 等）の関連チャンネル
  - Mastodon: 技術系インスタンスの関連投稿
- **検索言語**: 英語を優先。日本語の投稿は特に独自性が高い場合のみ採用
- **選定基準（以下のいずれかを満たす投稿を採用）:**
  1. **実践者の一次体験** — 実際に使った結果や発見をシェアしている（スクリーンショット・コード片・具体的な数値を含む）
  2. **議論に発展している** — リプライ・引用で実務者による具体的な補足・反論・検証が行われている
  3. **新しい手法・パターンの提示** — 既知の方法の紹介ではなく、独自の運用パターンやワークアラウンドを提示している
- **除外対象:** 感想のみの投稿、公式ドキュメントの引用のみ、フォロワー獲得目的の誇張投稿

### 2. AI・開発手法・ソフトウェア技術の動向 — 5件以上掲載

- AI コーディングツール・エージェント: Cursor、Windsurf、GitHub Copilot、Devin、OpenHands 等の新機能・比較・業界動向
- LLM・AI 基盤: 新モデルリリース、推論性能の進展、コスト変動、API の変更
- 開発手法: AI 駆動開発、スペック駆動開発、エージェント型開発等の実践報告・方法論の進化
- ソフトウェアアーキテクチャ: フロントエンド・フルスタックの新潮流、パフォーマンス最適化手法、セキュリティ動向
- 開発ツールチェーン: ビルドツール、パッケージマネージャ、CI/CD、テストフレームワーク等の重要な変更
- 技術トレンド: Web 標準の進化、ランタイム（Deno・Bun）動向、WebAssembly、Edge Computing、オブザーバビリティ等の最新動向

### 3. 経済ニュース — 5件以上掲載

- AI・テック企業の資金調達・M&A・IPO・業績発表
- AI 規制・法整備の動向（各国の AI 法案、EU AI Act の施行状況等）
- クラウドインフラ・SaaS の価格変動・サービス終了・統合
- 開発者の雇用市場・リモートワーク動向・スキル需要の変化
- テック業界に影響を与えるマクロ経済動向（為替・金利・半導体市場等）

### 4. 今日登場した新概念・新語彙

トピック 1〜3 の調査過程で**初めて登場した、または新たに命名された概念・用語・製品プリミティブ**を抽出する。パラダイムシフトは命名前の弱いシグナルとして現れるため、ここで日々の新語を構造化して蓄積し、weekly-research のメタ統合レイヤがそれを時系列で束ねて潮流を検知する材料にする。

- **抽出対象:**
  - 公式 changelog / リリースノートで新規導入された機能名・コマンド・プリミティブ（例: `/loop` `/goal` `workflow` `teammate` `ultracode` 等の**新しい名詞**）
  - 当事者・実践者が新たに定式化した手法・パターンの呼称（例: 「write loops」「Dynamic Workflows」「agentic laziness」）
  - 既存語の意味が変わった・再定義されたもの
- **除外対象:** 既に一般化した用語、単なる製品名の言及、過去レポートで既出の概念
- 各語彙について「初出ソース URL」「一言の定義」「過去レポートで関連語があれば言及」を記載する
- 該当が無ければ「該当なし」と明記する（セクション自体は省略しない）

## 完成度チェック（保存前に必ず実施）

保存前に自己点検し、不足があれば追加調査してから保存する:

1. 必読ソース（Claude Code changelog / Anthropic ブログ）を実際に確認したか。changelog に新バージョンがあればトピック2に必ず含めたか。
2. 当プロジェクトへの影響度「高」の項目（トークンコスト・課金・Claude Code 仕様変更等）を見落としていないか。
3. 手順0で把握した既報項目を誤って再掲していないか。
4. 各トピック5件以上の件数を満たしているか。

見落としがあれば追加調査して補ってから保存する。

## 出力ルール

- 更新があった項目のみ記載する（変更なしのトピックは省略）
- 各項目にソース URL を含める
- 影響度（高/中/低）を判定し、当プロジェクト（Tiptap ベースの Markdown エディタ）への影響があれば明記する
- 採用した記事・投稿については、以下の形式で内容の要約を記載する:
  - **タイトル / 投稿**（ソース URL）
  - **著者 / 媒体**: 著者名、所属、媒体名（SNS の場合はアカウント名・プラットフォーム）
  - **情報源レベル**: 一次情報 / 実践者の一次体験 / 二次情報（補足）
  - **要約**: 記事の主張・知見を3〜5行で要約。「何が書かれているか」だけでなく「なぜそう主張しているか」「どのような根拠・経験に基づいているか」も含める
  - **当プロジェクトへの示唆**: 該当する場合のみ記載
- トピック 4（今日登場した新概念・新語彙）は、レポート末尾（経済ニュースの後）に `## 今日登場した新概念・新語彙` セクションを設け、以下の表形式で出力する（weekly-research のメタ統合が機械的に集約するため形式を固定する）:

  | 語彙 / 概念 | 種別 | 初出ソース | 一言定義 | 関連既出語 |
  | --- | --- | --- | --- | --- |
  | write loops | 手法呼称 | https://... | プロンプトでなくオーケストレーションのループを書く設計思想 | Dynamic Workflows / Critic Loop |

  - **種別**: `製品プリミティブ`（新コマンド・新機能名）／ `手法呼称`（新たに命名された手法・パターン）／ `再定義`（既存語の意味変化）
  - 該当が無い場合は表の代わりに「該当なし」と1行記載する
- 結果を `<docsRoot>/report/daily-research/[今日の日付 YYYY-MM-DD]-daily-research.md` に保存する
- ファイルの先頭に以下のフロントマターを付与する（Web アプリの `/report` ページ表示用）:

```yaml
---
title: "Daily Technical Research - [今日の日付]"
date: "[今日の日付 YYYY-MM-DD]"
author: "Claude Code v[現在のCLIバージョン（`claude --version` で取得）]"
category: "Daily Research"
excerpt: "[当日の主要トピック3件程度を1文で要約。200文字以内]"
---
```

- フロントマターの後に `# Daily Technical Research - [今日の日付]` 見出しを付ける

## S3 アップロード

レポートファイルの保存後、`mcp__mcp-cms__upload_report` ツールで S3 にアップロードする。

1. `mcp__mcp-cms__upload_report` を呼び出す:
    - `filePath`: 保存したレポートファイルの絶対パス
2. レスポンスの `key` を確認し、アップロード結果をユーザーに報告する
3. アップロード失敗時はエラー内容を報告し、ローカルファイルは保持する

## Git コミット

S3 アップロード後、生成したレポートを docs リポジトリ（`<docsRoot>`）にコミットする。push はしない（ローカルコミットのみ）。

1. ファイル名を明示してステージする（`git add .` / `-A` / `--all` は禁止）:

   ```bash
   git -C <docsRoot> add report/daily-research/[今日の日付 YYYY-MM-DD]-daily-research.md
   ```

2. コミット前の3点確認（想定外があれば中断しユーザーに報告）:

   ```bash
   git -C <docsRoot> branch --show-current   # 想定ブランチか
   git -C <docsRoot> status --short           # 当該レポートのみステージか
   git -C <docsRoot> diff --cached --stat     # 身に覚えのない差分が無いか
   ```

3. コミットする。メッセージは Conventional Commits 形式 `docs(research): daily-research [今日の日付 YYYY-MM-DD]`。末尾に `Co-Authored-By: Claude Fable 5 <noreply@anthropic.com>` トレーラを付与する（現行既定モデル。モデル既定変更時は追随する）。
4. コミット結果（ハッシュ・1行サマリ）をユーザーに報告する。
