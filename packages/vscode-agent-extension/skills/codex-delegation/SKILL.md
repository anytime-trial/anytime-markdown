# Codex 委任ルール

更新日: 2026-07-11

Codex へ作業を委任する場合の共通ルール。**委任の媒体（チャット直接・プランファイル・スクリプト）に依らず、委任契約 6 点を満たしてから実行する。**

Claude Code は `codex exec` CLI を直接実行して Codex プロセスに作業を委任する（サブエージェントは未インストール。この環境は bwrap 不可のため `--dangerously-bypass-approvals-and-sandbox` が必須。実装例は `anytime-cross-review` スキルの `codex-review.cjs` パターン）。Codex は Claude のセッションコンテキストを継承しない。委任を確実に動作させるためには、**委任の境界と前提**を明示する必要がある。

## 適用形態

| 形態 | 適用方法 |
| --- | --- |
| 直接委任（チャット指示から即時実行） | 委任契約 6 点を委任プロンプトに織り込んで `codex exec` を実行する。6 点が揃わない場合は実行前にユーザーへ確認する |
| プランファイル内の委任タスク | 「プランファイル記述テンプレ」に従い、タスクごとに 6 点を記載する |
| スキル/スクリプト経由の定型委任 | スクリプト内プロンプトに 6 点を定数として埋め込む（例: `codex-review.cjs` は対象 diff・書式・read-only 制約を強制） |

## Claude / Codex 役割分担の原則

| 役割 | Claude（メイン） | Codex（サブ） |
| --- | --- | --- |
| プラン作成・進捗管理 | ○ | × |
| ブランチ作成・worktree 操作 | ○ | × |
| 実装（コード変更・ユニットテスト） | △ | ○ |
| ビルド・型チェック | △ | ○（実装と同セッション内） |
| 変更スコープ確認（`git diff --stat`） | ○ | × |
| コミット（3 点確認込み） | ○ | × |
| push・PR 作成・リリース | ○ | × |

> △ = 状況により Claude が実施することもある。

破壊的操作（push / merge / リリース）は **Claude 側で実施** し、Codex には委任しない（AGENTS.md「破壊的操作はユーザー承認必須」「リリースは production-release スキルでのみ」に基づく）。

## 委任契約 6 点（必須）

委任の媒体を問わず、各委任タスクに以下を 6 点すべて含める。1 つでも欠けると Codex が暗黙前提を取り違える。

| 項目 | 内容 |
| --- | --- |
| 1. 対象ファイル/ディレクトリ | 編集を許可するパス（複数可） |
| 2. 変更禁止範囲 | 触ってはいけないファイル/ディレクトリ |
| 3. 完了条件 | 何ができたら完了か（観測可能な条件） |
| 4. 検証コマンド | `npm run build -w <pkg>` / `npx jest <path> --maxWorkers=1` など。**コマンドが対象 package.json の** `scripts` **に存在し、依存パッケージが** `devDependencies` **に揃っているかを委任前に確認**（CLAUDE.md「プラン時 → 検証コマンドの実在確認義務化」参照）。Codex はサンドボックス内で実行されるため、未インストール devDep や未定義 script に対するエラーをそのまま返す |
| 5. 中断条件 | 何を観測したら作業を止めて状況報告するか（不能時の**正規の出口**。完了を装う・不能と判明した後に続けるの両方を禁じる）。汎用ルールは `references/stopping-rules-playbook.md` を同梱し、タスク固有の条件を追記する |
| 6. 委任プロンプト本文 | Codex に渡す具体的な指示（前提・実装方針・テスト要件） |

## プランファイル記述テンプレ

```markdown
## 委任ルール

このプランの実装タスクは Codex（`codex exec` CLI 直接実行。`--dangerously-bypass-approvals-and-sandbox` 必須。`anytime-cross-review` スキルの `codex-review.cjs` パターンを参照）に委任する。\
詳細は `codex-delegation` スキルに従う。

### 委任しない作業

- ブランチ作成・worktree 操作 → Claude が実施
- コミット（3 点確認込み）・push・PR 作成 → Claude が実施
- 破壊的操作（リリース・force push 等）→ Claude が実施

## タスク

### タスク 1: <タスク名>

- **対象ファイル**: `packages/<pkg>/src/foo.ts`
- **変更禁止**: `packages/<pkg>/src/bar.ts`
- **完了条件**: `foo()` の戻り値型が `Result<T, E>` に変更され、既存呼び出し元の型エラーが解消されている
- **検証**: `npx tsc --noEmit -p packages/<pkg>/tsconfig.json`
- **中断条件**: foo.ts の実態が上記前提と食い違う／検証コマンドが 2 回連続で同種の失敗をしたら、作業を止めて観測事実を報告する（playbook 準拠）
- **委任プロンプト**:
```

packages/<pkg>/src/foo.ts の foo() を Result 型に書き換える。

- 既存の throw を Err に置換、戻り値の正常系を Ok でラップする
- 呼び出し元の型エラーを解消（`packages/<pkg>/src/` 内のみ）
- 変更前に foo() の単体テストを TDD で追加する
- 実装後 `npx tsc --noEmit -p packages/<pkg>/tsconfig.json` で検証
- bar.ts は変更しないこと（責務分離のため）

```

### タスク 2: ...
```

> 直接委任（プランファイルなし）の場合も、上記タスク 1 と同じ 6 点構成の委任プロンプトを組み立ててから `codex exec` を実行する。委任内容が 3 ファイル以上の変更に及ぶ場合はプランファイル作成を優先する（CLAUDE.md「計画と実装」）。

## 委任プロンプトの書き方

委任プロンプト本文には以下を含める:

- **前提**: 既存実装の動作・関連コードの場所
- **方針**: どう書き換えるか（アルゴリズム・型シグネチャ・依存方向）
- **TDD**: 実装前にテストを書くか、書かない場合の理由
- **検証**: ビルド・型チェック・該当ユニットテストの実行
- **NG リスト**: 触ってほしくないファイル、避けてほしいパターン
- **中断条件（abstention）**: 止まるべき観測条件と、止まったときの報告形式。`references/stopping-rules-playbook.md` の汎用ルール（前提不整合・検証失敗の反復・完了条件の観測不能・環境制約）を同梱し、タスク固有の条件を足す。「完了を装う」「不能と判明した後に作業を続ける」の両方を明示的に禁じ、中断は失敗でなく**正規の完了形態**であると書く

Codex は AGENTS.md と `~/.codex/rules/*.md`（CLAUDE.md ルールのシンボリックリンク）を読むが、現在の Claude セッションで暗黙的に共有されている文脈（直前に直したバグ・進行中のリファクタ等）は継承しないため、委任プロンプトに明示的に書く。

## アンチパターン

以下の書き方は Codex の動作が曖昧になり、想定外の変更を引き起こす:

- 「実装は Codex で」の一文のみで終わる
- 対象ファイルを記載せず「機能 X を作る」だけ
- 完了条件を書かず「実装してテストも書いて」のみ
- 変更禁止範囲を書かない（Codex が広域を巻き込む恐れ）
- 中断条件を書かない（不能時に完了を装う・延々続ける圧力が構造的にかかる。虚偽報告の温床）
- Claude 側で実施すべき作業（コミット・push）を委任プロンプトに含める
- 検証コマンドの実在を確認していない（例: ルート package.json に無い `build` スクリプトを指示、`devDependencies` に無い `jest-environment-jsdom` を前提、`testMatch` 外の `.tsx` テストを追加など）。Codex のサンドボックスではホスト側の暗黙的なグローバルインストールが効かないため、委任前の事前確認が前提
- 直接委任で 6 点を口頭指示のまま省略する（チャットの曖昧な依頼をそのまま `codex exec` に流さない）

## チェックリスト

委任実行前（直接委任・プラン作成のいずれも）に以下を満たすこと:

- [ ] 各委任タスクに「対象 / 禁止範囲 / 完了条件 / 検証 / 中断条件 / プロンプト」6 点が揃っている
- [ ] 委任プロンプトに前提・方針・TDD・検証・NG リスト・中断条件（playbook 同梱）が含まれる
- [ ] 破壊的操作が委任プロンプトに含まれていない
- [ ] 検証コマンドが対象 package.json の `scripts` に存在し、必要 devDep が `devDependencies` に揃っていることを確認した
- [ ] （プランファイルの場合）委任ルール宣言と本ルールへのリンクがプラン冒頭にあり、委任しない作業（コミット・push 等）が列挙されている
