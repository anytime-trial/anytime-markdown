## 運用ルール

- 上限 約 4K トークン。超過しそうなら古い・汎用性の低いルールを統合・削除する
- 追記形式: `- <トリガ観測とルール 1 行>（出典: <セッション/事象>）`。委任の振り返り時に手動追記する
- 自動蒸留（Trail DB trajectory からの reflection → curation）は効果観測後に別提案で判断する
- 信頼境界の入力検証・データ損失防止・セキュリティ・アクセシビリティを「不能」と称して省く正当化には使わない（それらは中断でなく実装対象）

## 前提不整合

- 対象ファイルの実態（構成・型・命名）が指示の前提と食い違ったら、編集せず差異を報告する（出典: 初期整備）
- 指示された対象ファイル・シンボルが存在しなければ、類似物を勝手に代用せず報告する（出典: 初期整備）

## 検証失敗の反復

- 同一検証コマンドが 2 回連続で同種の失敗をしたら、3 回目を打たず失敗内容を報告する（出典: 初期整備）
- 検証コマンド自体が存在しない（script 未定義・devDep 不足）と判明したら、代替コマンドを自作せず報告する（出典: Codex 委任のアンチパターン実例）

## 完了条件の観測不能

- 完了条件がその環境で観測不能（実機・ブラウザ・外部サービス依存）と判明したら、「ユニット green」を根拠に完了と報告せず、観測できない旨を報告する（出典: jsdom 非再現領域の横断制約・実機 NG 3 例）

## 環境制約

- サンドボックス・権限・ネットワーク制約で必須操作が拒否されたら、迂回ハックを試みる前に報告する（出典: 初期整備）

## 委譲先の異常

- 委譲先が無応答・性能劣化で完了に至らないときは、同条件で再委任せず報告する（出典: Intelligent Delegation §4.4 の内部トリガ）

## 予算・不可逆性

- トークン予算・想定コストを超過したら、継続せず中断して方針を再評価する（出典: Intelligent Delegation §4.4 の内部トリガ）
- 不可逆・高重大度（`SKILL.md` 冒頭の定義）の失敗は自動リトライ・再委譲せず、即時停止してユーザーへエスカレーションする。可逆な失敗のみ前提修正後の再委任を許す（出典: 同 §4.4 の可逆性による応答分岐・MRM と同型）
